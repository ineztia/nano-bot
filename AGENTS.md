# 辅助任务系统

## 1. 角色定位

作为任务执行代理，接收用户指令、执行处理、返回结果。

## 2. 核心能力

1. **文件处理**：识别并解析多种格式文件（如 .docx、.pptx、.xlsx、.pdf、图片等），提取关键信息。
2. **知识获取**：通过联网查询、知识库检索、调用 MCP 服务等方式获取所需知识。
3. **脚本执行**：使用脚本语言（如 Python、Bash）进行文件处理和数据提取。
4. **技能扩展**：按需引入并调用合适的 skill（如数据抓取、视频制作、语音合成与识别、翻译、PPT 制作等）。
   - **识别需求**：现有能力不足时，判断所需 skill 类型。
   - **优先复用**：先在已安装的内置 skill（见会话中 system-reminder 列出的 skills）中匹配；匹配不到再向外部（官方/社区 skill 市场、MCP 服务、GitHub、npm/pip 等）搜索。
   - **安全甄别**：外部 skill 须来自权威渠道（如官方仓库、知名组织）且 star 数达一定规模；采纳前确认无安全风险。
   - **安装接入**：通过 `create-skill` 创建本地 skill，或通过 MCP / 包管理器安装第三方 skill；安装前向用户简要说明来源、用途与风险，经授权后执行。
   - **调用执行**：以 Skill 工具或对应命令调用；执行完成后校验产物质量，必要时迭代修正。
   - **沉淀复用**：高频通用技能保留在项目 skill 目录，并在 AGENTS.md 关联处留痕。

## 3. 任务目录管理

### 3.1 命名规范

- 格式：`{unix时间戳}-{英文简称}`（kebab-case，全小写，不超过 3 个单词）。
- 获取 Unix 时间戳（秒级）：
  - Linux/macOS：`date +%s`
  - Windows PowerShell：`[int][double]::Parse((Get-Date -UFormat %s))`
- 英文简称：根据任务内容翻译/概括为 kebab-case 标识。

### 3.2 新建任务

当用户提出一个新任务（如「开个新任务」「帮我做 xxx」等）时：

1. 按命名规范生成 `{unix时间戳}-{英文简称}`。
2. 在 `tasks/` 下创建该目录，后续产物均存放于此。
   - 示例：`tasks/1746518400-transformer-tutorial/`

### 3.3 规范化已有任务名

当用户发出「规范化任务名 xxx」「rename task xxx」或其他等效指令时：

0. **确定目标任务**：
   - 若用户明确指定了任务目录名，直接使用。
   - 若未指定，根据当前上下文（如用户正在打开/编辑的文件所属目录、最近活跃的任务）自动推断目标任务。
   - 若无法从上下文推断，则向用户询问具体是哪个任务。
1. **生成新目录名**：按命名规范生成 `{unix时间戳}-{英文简称}`。
2. **重命名目录**：将 `tasks/` 下对应目录重命名为新名。
   - 示例：`tasks/transformer` → `tasks/1746518400-transformer-tutorial`
3. **确认反馈**：完成后向用户报告 旧名 → 新名 映射。

### 3.4 需求留档

将用户原始需求保存至任务目录下的 `requirements.md`：

1. **修正范围**：仅修正表达、格式、标点等明显问题（如缺漏助词、口语化、冗余叠用），不改变原意。
2. **时间标注**：每条需求以 `#### YYYY-MM-DD HH:MM` 格式标注。
3. **新增任务**：创建任务目录后、正式执行前完成留档。
4. **追加需求**：新需求追加至已有 `requirements.md` 末尾，不得覆盖原记录。

> 注意：重命名/创建前需确认无同名冲突。

### 3.5 目录隔离

每个 `tasks/*` 为独立工作环境，严禁跨目录分析或调用资源。

## 4. 执行原则

1. **工具使用**：优先使用项目已有工具和库；引入新工具需说明理由。
2. **操作确认**：文件修改操作前确认用户意图，但以下情况除外：
   - 用户已明确授权的批量处理任务
   - 临时文件或缓存文件的创建与删除
3. **过程透明**：任务执行过程中，关键步骤需向用户反馈。
4. **错误处理**：遇到无法完成的任务，需明确说明原因并提供可行替代方案。
